### 3.3 Decision soundness (D1–D10)

Each decision must be (a) internally consistent with the matrix, (b) faithful
to the production code, and (c) non-breaking. In particular (post-Corrective-1):

- **D1 (result model, C7-strengthened):** AsyncCondition's direct
  `WaitOutcome` return is now justified primarily by the two-epoch protocol
  (Condition outcome latched before mandatory Mutex reacquire, returned only
  after ownership is restored), not merely by "users need to inspect the
  outcome". Is this rationale faithful to `condition.hpp` and the
  CONDITION-WAIT-PREPARE seam in `scheduler.cpp`?
- **D2 (WaitOutcome, C4-corrected):** fresh-per-epoch is enforced by the
  absorbing WaitNode state machine and the Detached-only registration
  precondition, NOT by the type system. Is this consistent with
  `wait_node.hpp` and `wait_queue.hpp::register_wait_locked`?
- **D3 (deadline precedence):** is resource-first precedence (already-due
  deadline + admissible resource → Woken, not Expired) actually implemented
  uniformly across Event/Semaphore/AsyncMutex/AsyncCondition/Queue?
- **D7 (cancellation):** is queue-identity-gated cancellation (wrong-object →
  false, no mutation) actually implemented uniformly?
- **D10 (E13 dependency, C2-corrected):** primitive cancel is now correctly
  recorded as best-effort per-queue cleanup, NOT a Select-level loser
  authority; E13 needs a parent/group claim ordering group-winner selection
  relative to irreversible primitive commit (e.g., AsyncMutex owner commit,
  Queue item commit). Is this consistent with the actual cancel semantics
  in `event.hpp`/`semaphore.hpp`/`async_mutex.hpp` and the commit points in
  `async_mutex.hpp:548-560` and `queue_port.hpp:728-739`?
